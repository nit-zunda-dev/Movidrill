# Movidrill — 企画書

> **動画を"観る"から"手を動かして身につける"へ。**
> YouTube自作動画×インタラクティブUIで、IT技術を体得する学習プラットフォーム。

---

## 1. プロダクト概要

### 1.1 コンセプト

Movidrill は、自身が作成したYouTube技術解説動画をWeb画面に組み込み、動画の進行とUI要素（コードエディタ、ターミナル、図解パネル等）がリアルタイムに連動するインタラクティブ学習プラットフォームである。

従来の「動画を観て終わり」という受動的な学習体験を、「動画と一緒に手を動かして正誤判定を受ける」能動的な体験に変革する。

### 1.2 名称の由来

**Movidrill** = **Movie（動画）** + **Drill（反復練習）**

動画を見ながら何度も手を動かして鍛える、というプラットフォームの本質を一語で表現。

### 1.3 ターゲットユーザー

| セグメント | ニーズ |
|:--|:--|
| プログラミング初学者 | 動画だけでは理解が浅い。手を動かして確認したい |
| 駆け出しエンジニア | インフラ・DB・セキュリティ等を体系的に学びたい |
| 技術系YouTuber | 自身の動画に付加価値をつけ、差別化したい |
| 企業の研修担当者 | 社内教育動画にハンズオン演習を付与したい |

### 1.4 競合との差別化

| 既存サービス | 特徴 | Movidrillとの違い |
|:--|:--|:--|
| Udemy / YouTube | 動画視聴のみ | Movidrillは動画＋実行環境＋自動判定が一体 |
| Progate / Codecademy | テキスト＋コード実行 | Movidrillは動画が主軸。映像と演習が同期 |
| LeetCode / HackerRank | 問題解決型 | Movidrillはチュートリアル型。段階的にステップ進行 |

### 2.3 YouTube IFrame API 連携仕様

```javascript
// 動画の埋め込みと制御
const player = new YT.Player('video-container', {
  videoId: 'YOUR_VIDEO_ID',
  events: {
    onReady: onPlayerReady,
    onStateChange: onPlayerStateChange
  }
});

// タイムスタンプ監視によるステップ同期
setInterval(() => {
  const currentTime = player.getCurrentTime();
  const step = findStepByTimestamp(currentTime);
  if (step && step.id !== state.currentStep) {
    player.pauseVideo();
    transitionToStep(step);
  }
}, 500);
```

### 3.1 QuerySync — SQL学習（✅ プロトタイプ実装済み）

| 項目 | 内容 |
|:--|:--|
| 概要 | SQL動画×インタラクティブエディタ。ブラウザ内でSQLを実行し即座にフィードバック |
| 技術スタック | sql.js（WebAssembly版SQLite） |
| UI構成 | 左: 動画 + ER図 + 課題カード / 右: SQLエディタ + 実行結果 + 判定 |
| 判定方式 | クエリ実行結果の行数・カラム数・値の一致を検証 |
| 難易度 | ★☆☆☆☆ |

**レッスン数・範囲の方針**

- **レッスン数**: **5本**（SELECT基本 ＋ JOIN ＋ サブクエリ・集計 ＋ DDL ＋ EXPLAIN）。プロトタイプ実装済み。
- **学習範囲**: **SQLの網羅ではなく「実務の入口まで」**を想定。扱うのは「読む（SELECT/JOIN/サブクエリ）」「作る（DDL）」「ちょっと速くする（EXPLAIN）」まで。INSERT/UPDATE/DELETE・トランザクション・インデックス設計・ビュー・ストアドプロシージャ・ウィンドウ関数などは対象外とする。

**レッスン一覧:** → [LESSONS.md#querysync](./LESSONS.md#querysync--sql学習)

**ロードマップ:**

## 6. YouTube動画の設計指針

### 6.1 動画制作時のルール

Movidrillと連動させるYouTube動画は、以下の構造で制作する。

1. **チャプター（タイムスタンプ）を明確に区切る** — 各チャプターが1つのステップに対応
2. **「ここで手を動かしてみましょう」のキュー** — 動画内で明確に演習タイミングを伝える
3. **解答の表示は次のチャプターの冒頭に** — ユーザーが自力で解いた後に確認できる構成
4. **1動画あたり3〜7ステップ** — 多すぎると離脱、少なすぎると物足りない

### 6.2 タイムスタンプ管理

```
00:00 - イントロ（学習ゴールの説明）
01:30 - Step 1: SELECT文の基本
03:45 - Step 2: カラム指定
06:00 - Step 3: WHERE句
08:30 - Step 4: ORDER BY
11:00 - Step 5: 集計関数
13:30 - まとめ
```

各タイムスタンプはレッスンJSONの `videoTimestamp` フィールドに秒単位で記録する。

|:--|:--|:--|:--|

| NetMap | トポロジ・ルーティング・パケット・NAT/FW | ✅ 4レッスン（予定）で網羅 | なし |
| HackLab | XSS/SQLi/CSRF/認可 | ✅ 4レッスン（予定）で網羅 | なし |

### 結論と追加で明示すべきタスク

- **シナリオの網羅性**: 各モジュールの「学習範囲」とレッスン一覧は対応しており、**シナリオの抜けはない**。Phase 2〜4 のモジュールは予定レッスンで目標達成に必要な範囲を記載済み。
- **目標達成のために要対応の抜け**:
  1. **YouTube動画連携**: 「3本以上の動画IDを設定し、埋め込み・タイムスタンプ同期を実装する」を Phase 1 のタスクとして明示する（または 4. ロードマップにチェック項目を追加済みのため、ここでは「KPI の YouTube 連携数は未達。上記タスク完了で解消」と記載済み）。
  2. **レッスンデータの外部JSON化**: 動画ID変更・ステップ追加をしやすくするため、Phase 1 に含まれており抜けなし。
  3. **APISync L3〜L5**: 拡充方針で「予定」として記載済み。実装優先度は Phase 2 以降で問題なし。

## 9. リスクと対策

| リスク | 影響度 | 対策 |
|:--|:--|:--|
| YouTube API の利用制限・仕様変更 | 高 | IFrame APIは安定。Data APIのquota管理を徹底 |
| ブラウザ内実行の制約（SQL以外） | 中 | WebContainerやService Workerの活用を検討 |
| レッスン作成コストが高い | 中 | レッスンJSON仕様を標準化し、作成ツールを早期構築 |
| モジュール間のUI統一性の維持 | 低 | 共通コンポーネントライブラリで一元管理 |
| セキュリティモジュールの安全性 | 高 | サンドボックスiframe + CSPヘッダーで隔離を徹底 |
